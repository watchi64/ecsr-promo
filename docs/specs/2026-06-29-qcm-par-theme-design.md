# Design · Système de QCM par thème (entraînement + examen)

Date : 2026-06-29
Projet : TP ECSR App (ecsr-promo)
Statut : design validé, à implémenter par lots

## 1. Contexte et objectif

L'app liste déjà les 106 entrées de la table `themes` (57 thèmes officiels + compétences TP ECSR + compétences REMC + notions) dans la vue Thèmes, avec statut « À faire / Fait » et date.

On ajoute un système de QCM rattaché à chaque thème, avec deux usages :

1. **QCM d'entraînement** : libre, illimité, correction immédiate, ne compte pas.
2. **QCM d'examen** : tirage aléatoire de N questions, une seule passe, correction à la fin. Le résultat est enregistré automatiquement comme **note** (sur 20) et alimente la moyenne de classe. L'examen n'est jouable que s'il a été **publié par un formateur**.

La vue Thèmes est enrichie pour afficher, par thème : l'accès aux QCM, l'état de publication de l'examen, **ma note**, **la date de mon examen**, et **la moyenne de la classe**. Un filtre « mes points faibles » trie les thèmes par ma note croissante pour refaire les entraînements.

## 2. Décisions cadrées (validées avec l'utilisateur)

| Sujet | Décision |
|---|---|
| Stockage des questions | Base de données Supabase + éditeur in-app (formateurs). Amorcé avec 2 QCM fournis par Hocine. |
| Entraînement vs examen | **Même banque** de questions par thème. Examen = **tirage aléatoire** de N questions. |
| Réglage examen | Chaque QCM porte son **N** (nombre de questions tirées) et son **seuil de réussite** sur 20 (affiché « réussi / à retravailler », purement informatif). |
| Enregistrement de la note | **1 passage officiel** par stagiaire. Score converti **/20** → tentative `examen` + (si thème numéroté) miroir dans `evaluations`. Re-passage uniquement si un formateur réinitialise. |
| Visibilité | Chaque stagiaire voit **sa note** + **la moyenne de classe**. Les notes individuelles des autres ne sont pas exposées dans la vue Thèmes. |
| Format des questions | QCU (1 bonne réponse) mais modèle générique (`is_correct` par option) compatible QCM multi. Option « Je ne sais pas. » = option jamais correcte. |
| Périmètre de départ | **Lot 1 d'abord** (socle + import + entraînement), validé en preview locale, puis lots 2 et 3. |

## 3. Architecture des données

Source de vérité des notes QCM = la table `qcm_attempts`. La matrice Notes (`evaluations`) ne reçoit qu'un **miroir** des examens rattachés à un thème numéroté (car elle indexe les notes « Thème » par `theme_numero` 1 à 57, et les notes « Compétence » par la table `competences` à 6 lignes ; les compétences REMC, `numero = null`, n'y rentrent pas proprement).

### 3.1 Tables (DDL cible, forward-compatible pour les 3 lots)

```sql
-- Un QCM par ligne themes (réutilise les 106 entrées existantes)
create table qcm (
  id                 bigint generated by default as identity primary key,
  theme_id           integer not null references themes(id) on delete cascade,
  titre              text not null,
  description        text,
  published          boolean not null default false,   -- examen en ligne ?
  published_by_email text,
  published_at       timestamptz,
  exam_nb_questions  integer,                           -- N tirées (null = toutes)
  exam_pass_20       numeric not null default 12,       -- seuil /20 (informatif)
  created_by_email   text,
  created_at         timestamptz not null default now(),
  updated_at         timestamptz not null default now(),
  unique (theme_id)                                     -- 1 QCM par thème (v1)
);

create table qcm_questions (
  id          bigint generated by default as identity primary key,
  qcm_id      bigint not null references qcm(id) on delete cascade,
  section     text,            -- sous-compétence : « Embrayage », « Freinage »…
  enonce      text not null,
  explication text,            -- affichée en correction (optionnel)
  ordre       integer not null default 0,
  created_at  timestamptz not null default now()
);

create table qcm_options (
  id          bigint generated by default as identity primary key,
  question_id bigint not null references qcm_questions(id) on delete cascade,
  texte       text not null,
  is_correct  boolean not null default false,
  ordre       integer not null default 0
);

-- Source de vérité des notes (examen ET entraînement)
create table qcm_attempts (
  id          bigint generated by default as identity primary key,
  qcm_id      bigint not null references qcm(id) on delete cascade,
  stagiaire_id integer not null references stagiaires(id) on delete cascade,
  mode        text not null check (mode in ('entrainement','examen')),
  score       integer not null,        -- bonnes réponses
  total       integer not null,        -- questions posées (= N pour l'examen)
  note_20     numeric not null,        -- round(score/total*20, 1)
  answers     jsonb,                   -- snapshot {question_id: option_id} (révision/audit)
  started_at  timestamptz,
  finished_at timestamptz not null default now()
);

-- Index utiles
create index on qcm_questions (qcm_id, ordre);
create index on qcm_options (question_id, ordre);
create index on qcm_attempts (qcm_id, stagiaire_id, mode);
```

Note : un seul examen officiel par stagiaire est garanti côté logique applicative + une contrainte partielle pourra être ajoutée au lot 2 (`unique (qcm_id, stagiaire_id) where mode = 'examen'`), la réinitialisation supprimant la tentative.

### 3.2 RLS (principes, détail au moment de la migration)

- `qcm`, `qcm_questions`, `qcm_options` : **lecture** pour tout utilisateur authentifié (l'entraînement doit être visible) ; **écriture** réservée prof/admin (via `user_profiles.role in ('prof','admin')` ou `is_admin`).
- `qcm.published` ne change que par prof/admin.
- `qcm_attempts` : un stagiaire **lit et écrit uniquement ses propres lignes** (`stagiaire_id` = son `user_profiles.stagiaire_id`) ; prof/admin lisent tout (moyennes, réinitialisation) et peuvent supprimer une tentative (réinit).
- La **moyenne de classe** sera calculée via une **vue / fonction RPC** agrégée (`avg(note_20)` par `qcm_id`, mode examen) pour ne pas exposer les notes individuelles des autres aux stagiaires.

## 4. Flux fonctionnels

### 4.1 Entraînement (tout le monde, illimité)

1. Ouverture depuis la modale thème, plein écran, look des QCM statiques mint existants (`qcm/qcm_matrice_gde_remc.html` comme référence visuelle).
2. Toutes les questions du QCM, **ordre mélangé**.
3. **Correction immédiate** par question (vert/rouge + explication), score live, barre de progression.
4. À la fin : écran résultat + recap. Tentative enregistrée en `mode = entrainement` (historique perso), **non comptée**, n'alimente pas la moyenne.

### 4.2 Examen (si `published = true`)

1. Pré-écran : rappel « 1 seule passe, N questions, seuil X/20 ». Bloqué si une tentative `examen` existe déjà (sauf réinit formateur).
2. **Tirage aléatoire** de `exam_nb_questions` questions (défaut = toutes), ordre mélangé. **Pas de correction avant la fin.**
3. À la validation :
   - `note_20 = round(score / total * 20, 1)`.
   - Insert `qcm_attempts` (`mode = examen`, snapshot `answers`).
   - Si `themes.numero is not null` → **upsert `evaluations`** (type « Thème », `theme_numero`, `theme_titre`, `note = note_20`, `note_max = 20`, `date_eval = current_date`, `observation = 'QCM examen'`) pour apparaître aussi dans la matrice Notes.
   - Si thème non numéroté (REMC, notion) → pas de miroir `evaluations` en v1 (note visible dans la vue Thèmes via `qcm_attempts`). Extension possible plus tard.
4. Écran résultat : note /20, label **« Réussi »** si `note_20 >= exam_pass_20` sinon **« À retravailler »**, recap des questions (avec les bonnes réponses).

### 4.3 Publication / réinitialisation (formateur)

- Publier / dépublier l'examen depuis la modale thème (toggle `published` + `published_by_email` / `published_at`).
- Réinitialiser la tentative d'un stagiaire (suppression de sa ligne `examen` + suppression du miroir `evaluations` correspondant si présent) pour autoriser un nouveau passage.

## 5. Permissions (récap)

| Action | Stagiaire | Formateur / Admin |
|---|---|---|
| Entraînement | oui | oui |
| Examen (si publié) | oui, 1 fois | oui |
| Créer / éditer questions, marquer les bonnes réponses | non | oui |
| Publier / dépublier l'examen | non | oui |
| Réinitialiser la tentative d'un stagiaire | non | oui |
| Voir sa note + moyenne de classe | oui | oui |
| Voir les notes individuelles des autres | non | oui |

## 6. Vue Thèmes enrichie (mobile-first)

Sur chaque ligne thème, en complément de N° / Thème / Statut / Fait le / Notes :

| Élément | Affichage | Visible par |
|---|---|---|
| Entraînement | bouton ▶ (si le QCM a au moins 1 question) | tous |
| Examen | badge « En ligne » (publié) / « Brouillon » | tous (action publier : formateur) |
| Ma note | `14/20` ou `—` | le stagiaire connecté |
| Date examen | date de ma tentative examen | le stagiaire connecté |
| Moyenne classe | `12,4/20` (avg des `note_20` examen) | tous |

- Filtre **« Mes points faibles »** : ne garde que les thèmes où j'ai une note d'examen, triés par ma note croissante, pour relancer l'entraînement.
- Conventions UI respectées : mint éditorial unique, « Formateur » (jamais « Prof »), pas d'em-dash, couleurs notes pêche → vert (paliers 8/12/16), pas de rouge vif.
- Player et éditeur ouverts depuis la modale thème (qui contient déjà le placeholder « Contenu pédagogique à venir : cours, QCM »).

- `js/views/notes.js` : aucune refonte ; le miroir `evaluations` réutilise le format type « Thème » déjà géré (clé `theme_numero`).

## 8. Amorçage des 2 QCM d'Hocine

Fichiers fournis (Downloads, 2026-06-29) :

| Fichier | Rattachement `themes` | Questions | Sections |
|---|---|---|---|
| `Evaluation C1 type QCU.docx` | `REMC C1 — Maîtriser le maniement du véhicule` (id 80) | 20 | Démarrer/s'arrêter, Tenir le volant, Boîte de vitesses, Doser frein/accélérateur, Regarder/avertir |
| `QCU QR CODE Utilisation rationnelle du véhicule.docx` | **à confirmer** (REMC C4 conduite économique, ou thème numéroté éco-conduite) | 12 | Embrayage, Freinage, Boîte de vitesses |

- Chaque question a 4 vraies options + « Je ne sais pas. ».
- **Les bonnes réponses ne sont PAS marquées dans les .docx** : import des questions + options avec `is_correct = false` partout, puis **Hocine valide les bonnes réponses dans l'éditeur**. Option : pré-cocher des suggestions clairement étiquetées « à valider » pour accélérer (ne jamais présenter une suggestion comme officielle).

## 9. Découpage en lots

### Lot 1 · Socle + entraînement (périmètre d'implémentation immédiat)

- Migration : tables `qcm`, `qcm_questions`, `qcm_options`, `qcm_attempts` + RLS de base.
- Fonctions `db.js` de lecture + insert attempt (entraînement).
- Import des 2 QCM d'Hocine (réponses à valider).
- Player **entraînement** plein écran depuis la modale thème.
- Indicateur « QCM dispo » sur la ligne thème.
- Livraison : utilisable tout de suite pour réviser. Validation preview locale.

### Lot 2 · Examen + note

- Mode examen (tirage N, une passe, seuil) + écran résultat.
- Écriture `qcm_attempts` (examen) + miroir `evaluations` pour thèmes numérotés.
- Publication / dépublication par formateur, réinitialisation.
- Contrainte d'unicité examen.

### Lot 3 · Vue Thèmes enrichie + éditeur

- Colonnes ma note / date examen / moyenne classe.
- Filtre « mes points faibles ».
- Éditeur complet de questions/options/sections (formateur).

## 10. Points ouverts / risques

- **Bonnes réponses à valider** par Hocine (32 questions). Bloquant pour que l'examen ait du sens ; l'entraînement peut démarrer avec corrections provisoires si étiquetées.
- **Rattachement du 2e QCM** (« Utilisation rationnelle ») à confirmer : REMC C4 ou un thème numéroté.
- **Calcul moyenne de classe sans fuite** : via vue/RPC agrégée (les stagiaires ne doivent pas pouvoir lire les `note_20` individuelles des autres).
- **Arrondi note_20** : `round(score/total*20, 1)` ; affichage entier si rond.
- **`unique (theme_id)` sur `qcm`** : un seul QCM par thème en v1. Si besoin de plusieurs QCM par thème plus tard, lever la contrainte.
